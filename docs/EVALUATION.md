### Known caveats

- The seed dataset is intentionally small (~210 examples); the spec calls for ~500 before production. Expect F1 to climb as the dataset grows past ~300 labelled utterances.
- The `oos` class is the hardest — prompt-injection-style probes ("ignore previous instructions") need more examples.
- We deliberately ship a **rule-based fallback** (`RuleBasedClassifier`) that caps confidence at 0.75. If the transformer model is missing, logs will show `intent_using kind=rule_based`.

## 2. Entity extraction

Target: **F1 ≥ 0.85** on mood / genre / year / runtime slots.

Entity extraction is currently gazetteer + regex based (see `app/nlp/entity_extractor.py`), not a learned model. That choice is explained in the module docstring — with ~500 examples we can't train a full NER model to production quality, and a curated lookup gives tight precision on the vocabulary we care about. The test suite in `backend/tests/unit/test_entity_extractor.py` covers 18 cases across moods (`sad`, `happy`, `stressed`), genres (with longest-match-wins for `"science fiction"` vs `"fiction"`), decade parsing (`1980s` → `[1980, 1989]`), age, runtime, min-rating, and language codes.

A proper held-out evaluation needs an additional annotated utterance file — recommend creating `ml/data/entities_eval.jsonl` with ≥100 utterances, each tagged with expected entity dict, then scoring exact-match F1 per slot.

## 3. Recommendation quality

Target: **NDCG@5 ≥ 0.7** against a held-out set.

### How to evaluate

The spec calls for NDCG@5 — this requires (query, relevant-items) pairs. Two paths:

1. **Offline, using real user data** — once the system has been live and accumulated ≥1000 `ratings` rows, we can replay user sessions: for each `recommend` turn, check how many of the 5 returned items the user subsequently liked. This produces an NDCG@5 score per intent per week.
2. **Synthetic evaluation set** — hand-label 50 canonical queries (e.g. "feel-good 90s comedy") with 10–20 expected movies each, run the retriever, compute NDCG@5 against the graded relevance.

Neither evaluator is shipped in the initial repo — they are follow-on work once real usage produces signal. The recommender's unit tests (`backend/tests/unit/test_ranker.py`) cover:

- MMR diversifies near-duplicates (relevance vs diversity trade-off)
- Freshness boost applies to recent films
- Low-vote-count films are penalized

## 4. Latency

Target: **p95 chat turn < 2s**.

Run the Locust load test:

```bash
make dev
make seed
make load-test
# In the web UI: spawn 50 users, 10/s, target http://localhost:8000
```

Then read the p95 from Grafana → **CineBot — Service Overview** → p95 latency panel.

### Expected breakdown (per turn)

| Stage | Expected | Notes |
| --- | ---: | --- |
| NLP (intent + NER + embedding) | 50–120ms | Embedder is the bottleneck; first-turn cold start ~400ms |
| Retrieval (pgvector HNSW) | 5–20ms | Scales sub-linearly with catalog size |
| CF scoring | 5–30ms | Only active for users with ≥5 ratings |
| MMR rerank | 1–5ms | Pure in-memory |
| Explanation (template only) | <1ms | |
| Explanation (LLM-polished) | 300–1500ms | Primary p95 contributor when LLM polish is on |
| DB persistence | 5–15ms | |

Template-only path should comfortably hit sub-500ms p95. LLM-polish path depends on provider — OpenAI GPT-4o-mini runs ~500ms; Ollama + Llama 3.1 8B on CPU runs ~2–4s and should only be used for batch/offline polish, not the hot path.
